# Man-in-the-Middle Attack Vulnerability
During early development of my application, I discovered a critical security issue: data transmitted between the client and server was not encrypted. This made the application vulnerable to man-in-the-middle (MITM) attacks, where an attacker can intercept and read sensitive communication.
<img width="3838" height="2023" alt="wireshark1" src="https://github.com/user-attachments/assets/c04467e4-10de-46b5-8f0d-f16b020a03ef" />
Using a network analysis tool like Wireshark, I was able to easily eavesdrop on the traffic. For example, when a client sends "secret info" to the server and the server responds, the data is visible in plain text. This is clearly unacceptable for any secure application.
# Solution
To address this vulnerability, I implemented TLS (Transport Layer Security) encryption. TLS ensures that even if network traffic is intercepted, the data remains unreadable without the appropriate decryption keys. For development purposes, I generated a self-signed certificate. While self-signed certificates are not trusted by browsers, they still use modern cryptographic algorithms and provide strong encryption. The key difference is that they are not issued by a trusted Certificate Authority (CA), which is why browsers display warnings. For local development, however, they are perfectly sufficient.

# Generating the Certificate

```bash
openssl req -x509 -newkey rsa:2048 \
-keyout server.key \
-out server.crt \
-days 365 \
-nodes \
-subj "/CN=localhost"
```
This command generates two files:

`server.key` — The private key, used by the server to decrypt incoming data. This file must remain secret and should never be shared.
`server.crt` — The certificate, which contains the server’s public key and is used by clients to verify the server’s identity.

TLS Handshake Overview

After enabling TLS, the communication process changes significantly:

A standard TCP three-way handshake establishes the connection.
The client sends a Client Hello message, initiating the TLS handshake.
The server responds with a Server Hello.
The server provides its certificate, which includes its public key.
The client (in this case, configured to trust the self-signed certificate) accepts it.
Both parties agree on a TLS version and exchange cryptographic parameters.
A secure, encrypted channel is established.

When observing this process in Wireshark after implementing TLS, the payload data is no longer readable, confirming that the communication is now properly encrypted.
